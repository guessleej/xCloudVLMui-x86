## 1. 整合策略總覽

### 1.1 漸進式整合原則

xCloudVLMUI 採用「漸進式整合」策略，確保：

1. **不中斷現有作業**：新整合不影響已在運行的生產系統
2. **可回滾設計**：每個整合步驟都可獨立停用
3. **介面穩定性**：外部系統透過標準 API 整合，內部變更不影響外部
4. **功能旗標（Feature Flag）**：新功能可透過 SystemSettings 開關

### 1.2 整合層次架構

```
Level 1：感測器資料整合（MQTT/Modbus）── 已完成 ✅
Level 2：LLM 推論引擎整合（llama.cpp）── 已完成 ✅
Level 3：知識庫整合（ChromaDB RAG）───── 已完成 ✅
Level 4：多感測器協議擴展（Modbus/REST）─ v1.2 規劃
Level 5：外部 MES/ERP 系統整合 ──────── v2.0 規劃
Level 6：多站點集中管理 ─────────────── v3.0 規劃
```

### 1.3 版本里程碑

```
v1.0.0 (2026-Q1)  基礎 VLM 視覺檢測
    │
v1.1.0 (2026-Q2)  MQTT 感測器監控 + Event Center + 告警 CRUD
    │
v1.2.0 (2026-Q3)  Modbus/REST 感測器 + 多 LLM 支援 + 告警通知
    │
v2.0.0 (2026-Q4)  MES 整合 + PostgreSQL + 多站點
    │
v3.0.0 (2027-H1)  雲端混合部署 + 聯邦學習
```

## 3. v1.2.0 整合規劃

### 3.1 Modbus TCP 感測器整合

**目標：** 支援透過 Modbus TCP 直接讀取 PLC/感測器資料

**實作計畫：**

```
新增檔案：
  backend/adapters/modbus_adapter.py
  backend/routers/modbus.py
  backend/schemas/modbus_schemas.py

新增 API：
  GET  /api/modbus/configs           # Modbus 配置管理
  POST /api/modbus/configs
  POST /api/modbus/poll              # 手動觸發輪詢
  GET  /api/modbus/status            # 輪詢狀態

設定項目（SystemSettings）：
  modbus.enabled: false
  modbus.poll_interval_seconds: 30
```

**Modbus 輪詢架構：**

```python
# 背景任務（APScheduler）
@scheduler.scheduled_job('interval', seconds=30, id='modbus_poll')
async def poll_modbus_sensors():
    configs = await modbus_config_repo.get_enabled()
    for config in configs:
        reading = await modbus_adapter.read_register(
            device_id=config.device_id,
            register_address=config.register_address,
            sensor_type=config.sensor_type,
            scale=config.scale,
            unit=config.unit
        )
        await sensor_repo.create(reading)
        # 閾值判斷（同 MQTT）
```

### 3.3 多 LLM 支援

**目標：** 支援透過 SystemSettings 切換 LLM 後端，無需重啟服務

| LLM 後端 | 優點 | 適用場景 |
|----------|------|----------|
| llama.cpp | 本機推論，離線可用 | 標準 Jetson 部署 |
| Ollama | 管理介面更友善 | 開發/測試 |
| OpenAI Compatible | 雲端算力，速度快 | 有網路連線的廠區 |

**切換機制：**

```python
# SystemSettings 中新增
{
    "key": "llm.backend",
    "value": "llamacpp",  # llamacpp | ollama | openai
    "description": "LLM 推論後端選擇"
}

# 熱切換（無需重啟）
@app.on_event("startup")
async def setup_llm():
    llm_adapter = get_llm_adapter()  # 從 settings 讀取
    app.state.llm = llm_adapter

# PUT /api/settings 更新後自動重新初始化 LLM adapter
```

## 4. v2.0.0 整合規劃

### 4.1 MES（製造執行系統）整合

**目標：** 與工廠 MES 雙向資料交換

```
整合方向：
  MES → xCloud：設備維護工單下達
  xCloud → MES：VLM 檢測報告上傳、告警通知

整合方式：
  REST API（首選）
  OPC-UA（設備直連備選）
  資料庫直連（最後手段，不建議）

資料映射：
  Equipment ↔ MES 設備主檔
  InspectionReport ↔ MES 維護工單
  EquipmentAlert ↔ MES 異常通報單
```

### 4.2 PostgreSQL 遷移

**目標：** 支援多並發寫入、更完善的備份機制

```yaml
# 遷移後 docker-compose.yml 新增服務
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: xcloudvlm
      POSTGRES_USER: xcloud
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U xcloud"]
```

**遷移腳本：**

```bash
# scripts/migrate-to-postgres.sh
# Step 1: 備份 SQLite
cp data/xcloudvlm.db data/xcloudvlm.db.migration-backup

# Step 2: 匯出資料
python scripts/export_sqlite.py --output /tmp/migration.json

# Step 3: 啟動 PostgreSQL
docker compose up -d postgres

# Step 4: 匯入資料
python scripts/import_postgres.py --input /tmp/migration.json

# Step 5: 切換後端設定
sed -i 's/DATABASE_URL=sqlite/DATABASE_URL=postgresql/' backend/.env
docker compose restart backend
```

## 5. 工廠系統整合指南

### 5.1 MQTT 整合步驟

```
Step 1：在設備管理頁新增設備（設備 ID 需與 MQTT device_id 對應）

Step 2：進入設定 → MQTT 設定，新增 MQTT 配置：
  - Broker 位址：mosquitto（預設）或外部 Broker IP
  - Topic 格式：xcloud/{device_id}/{sensor_type}
  - 閾值設定：根據感測器規格設定上/下限

Step 3：感測器裝置設定 MQTT 發布：
  Broker: {Jetson IP}:1883
  Topic: xcloud/pump001/temperature
  Payload: {"value": 72.5, "unit": "°C"}
  QoS: 1

Step 4：在 Event Center 確認資料接收
Step 5：在設備詳情 Drawer 確認感測器讀值顯示
```

### 5.2 API 整合步驟

```
Step 1：確認 Backend API 可從外部訪問
  預設位址：http://{Jetson IP}/api

Step 2：建立設備（若未透過 UI 建立）
  POST http://{Jetson IP}/api/equipment
  Content-Type: application/json
  {
    "name": "Pump-001",
    "device_type": "centrifugal_pump",
    "location": "廠房 A-1F"
  }

Step 3：直接推送感測器評分（不使用 MQTT 時）
  POST http://{Jetson IP}/api/vhs/{equipment_id}
  {
    "score": 85.5,
    "source": "manual",
    "notes": "現場目測"
  }

Step 4：查詢告警（供 MES 或 SCADA 輪詢）
  GET http://{Jetson IP}/api/alerts?resolved=false&severity=critical
```

### 5.3 整合驗證清單

```
□ MQTT Broker 可從感測器裝置連線
□ 感測器資料出現在 Event Center（INFO level，source=mqtt）
□ 超閾值告警出現在 AlertBanner
□ VLM 推論可正常觸發（POST /api/equipment/{id}/inspect）
□ API /health 回傳 {"status": "healthy"}
□ 所有四個 Pipeline Stage 都能正常完成
```

## 7. 整合風險管理

### 7.1 整合風險矩陣

| 風險 | 影響 | 概率 | 緩解措施 |
|------|------|------|----------|
| MQTT 連線不穩定 | 資料遺失 | 中 | QoS=1 重傳 + 本機緩衝 |
| LLM 推論逾時 | 服務降級 | 低 | 30 秒 timeout + 重試機制 |
| 感測器資料格式不符 | 解析錯誤 | 中 | 彈性 Payload 解析 + 原始資料記錄 |
| 外部系統 API 版本衝突 | 整合中斷 | 低 | API 版本控制 + 向下相容 |
| 網路分段阻擋 | 無法連線 | 高 | 確認防火牆規則（Port 1883, 8080） |

### 7.2 降級策略（Graceful Degradation）

```
MQTT 不可用：
  → 繼續接受手動 API 評分（POST /api/vhs）
  → 告警橫幅顯示 "MQTT 連線中斷"
  → syslog 記錄連線失敗事件

LLM 不可用：
  → 感測器監控繼續運行
  → VLM 推論請求回傳 503 Service Unavailable
  → 儀表板顯示 "VLM 推論服務暫停"

ChromaDB 不可用：
  → 停用 RAG 知識整合（Pipeline 跳過 knowledge_integration）
  → VLM 推論改為純視覺模式（不含知識庫輔助）
```
