## 1. KPI 框架概述

### 1.1 KPI 分類架構

```
xCloudVLMUI KPI Framework
├── L1：系統效能（System Performance）
│   ├── 可用性（Availability）
│   ├── 回應時間（Response Time）
│   └── 吞吐量（Throughput）
│
├── L2：業務價值（Business Value）
│   ├── 設備健康改善（VHS Improvement）
│   ├── 非計畫停機減少（Unplanned Downtime Reduction）
│   └── 維護成本降低（Maintenance Cost Reduction）
│
├── L3：維護效率（Maintenance Efficiency）
│   ├── 檢測準確率（Inspection Accuracy）
│   ├── 告警處理時效（Alert Response Time）
│   └── 預測維護命中率（Predictive Maintenance Hit Rate）
│
└── L4：感測器監控（Sensor Monitoring）
    ├── 資料完整率（Data Completeness）
    ├── 告警誤報率（False Positive Rate）
    └── 閾值覆蓋率（Threshold Coverage）
```

### 1.2 KPI 評估週期

| 評估週期 | 涵蓋 KPI | 彙報對象 |
|----------|----------|----------|
| 每日（自動） | 系統可用性、API 回應時間 | 工程師 Dashboard |

## 2. 系統效能 KPI

### 2.1 服務可用性（SLA）

| 指標 | 定義 | 目標 | 量測方式 |
|------|------|------|----------|
| **整體可用性** | 服務正常運行時間 / 總時間 | ≥ 99.5% | `/api/health` 輪詢 |
| **LLM 可用性** | llama.cpp 回應成功率 | ≥ 99% | `GET :8080/health` |
| **MQTT 連線穩定性** | MQTT 連線持續時間比率 | ≥ 99.9% | `GET /api/mqtt/status` |

```
月度計算：
可用性 = (總分鐘數 - 中斷分鐘數) / 總分鐘數 × 100%

目標 99.5% → 允許停機：≤ 3.65 小時/月
目標 99.9% → 允許停機：≤ 43.8 分鐘/月
```

### 2.2 API 回應時間

| 端點類型 | P50 目標 | P95 目標 | P99 目標 |
|----------|----------|----------|----------|
| 讀取類（GET） | < 50ms | < 200ms | < 500ms |
| 寫入類（POST/PUT） | < 100ms | < 500ms | < 1000ms |
| VLM 推論（POST /inspect） | < 2000ms | < 8000ms | < 15000ms |
| 串流對話（Chat Stream） | 首 token < 1000ms | — | — |

**量測方法：** 從 `syslog.db` 的 `duration_ms` 欄位計算分位數

```sql
-- P95 回應時間（過去 24 小時）
SELECT 
    path,
    method,
    COUNT(*) as requests,
    ROUND(AVG(duration_ms), 1) as avg_ms,
    ROUND(
        (SELECT duration_ms FROM syslog s2 
         WHERE s2.path = s1.path 
         ORDER BY duration_ms 
         LIMIT 1 OFFSET CAST(COUNT(*) * 0.95 AS INT)
        ), 1
    ) as p95_ms
FROM syslog s1
WHERE timestamp > datetime('now', '-24 hours')
GROUP BY path, method
ORDER BY p95_ms DESC;
```

## 3. 業務價值 KPI

### 3.1 設備健康評分（VHS）改善

| 指標 | 定義 | 基準值 | 目標 |
|------|------|--------|------|
| **平均 VHS 分數** | 所有設備 VHS 平均值 | 建置前人工估計 | ≥ 75 |
| **低分設備比率** | VHS < 60 的設備比率 | — | ≤ 10% |
| **VHS 資料覆蓋率** | 有 VHS 記錄的設備比率 | 0% | ≥ 90% |

```sql
-- 月度 VHS 趨勢
SELECT 
    strftime('%Y-%m', recorded_at) as month,
    AVG(score) as avg_vhs,
    MIN(score) as min_vhs,
    COUNT(DISTINCT equipment_id) as equipment_count,
    SUM(CASE WHEN score < 60 THEN 1 ELSE 0 END) as low_score_count
FROM vhs_readings
GROUP BY month
ORDER BY month DESC
LIMIT 12;
```

### 3.2 非計畫停機指標

| 指標 | 定義 | 目標 |
|------|------|------|
| **MTBF 改善率** | 平均故障間隔時間提升 | ≥ 20% (vs 建置前) |
| **MTTR 縮短率** | 平均修復時間縮短 | ≥ 30% (vs 建置前) |
| **緊急維修單比率** | 緊急 vs 預防性維護比 | ≤ 20% |

**量測基準建立：**
1. 系統上線前 3 個月：收集歷史停機紀錄作為 Baseline
2. 系統上線後每月：比對停機事件與 VHS 告警提前量

### 3.3 VLM 推論效益

| 指標 | 定義 | 目標 |
|------|------|------|
| **月度 VLM 檢測次數** | 成功完成四段式推論次數 | ≥ 每設備 4 次/月 |
| **推論成功率** | 成功 / 總觸發次數 | ≥ 95% |
| **人工複驗吻合率** | VLM 判斷與人工複驗一致 | ≥ 80% |

## 4. 維護效率 KPI

### 4.1 告警處理時效

| 等級 | 回應時間目標 | 解除時間目標 |
|------|-------------|-------------|
| `critical` | ≤ 15 分鐘 | ≤ 4 小時 |
| `warning` | ≤ 2 小時 | ≤ 24 小時 |
| `info` | ≤ 24 小時 | ≤ 72 小時 |

```sql
-- 告警平均解除時間（按等級）
SELECT 
    severity,
    COUNT(*) as total_alerts,
    AVG(
        JULIANDAY(resolved_at) - JULIANDAY(created_at)
    ) * 24 * 60 as avg_resolution_minutes,
    SUM(CASE WHEN resolved = 0 THEN 1 ELSE 0 END) as unresolved_count
FROM equipment_alerts
WHERE created_at > datetime('now', '-30 days')
GROUP BY severity;
```

### 4.2 預測維護命中率

| 指標 | 定義 | 計算方式 |
|------|------|----------|
| **True Positive（TP）** | VHS 告警後確認有問題 | 維護工單確認 |
| **False Positive（FP）** | VHS 告警但設備正常 | 人工複驗結果 |
| **命中率** | TP / (TP + FP) | 目標 ≥ 70% |
| **漏報率** | 設備故障前無告警 | 目標 ≤ 15% |

### 4.3 知識庫使用效益

| 指標 | 目標 |
|------|------|
| 知識文件數量 | ≥ 每設備類型 3 份文件 |
| RAG 查詢命中率（Top-5 相關） | ≥ 80% |
| 維護人員知識查詢滿意度 | ≥ 4.0 / 5.0 |

## 5. 感測器監控 KPI

### 5.1 資料完整率

| 指標 | 定義 | 目標 |
|------|------|------|
| **資料完整率** | 實際接收資料點 / 預期資料點 | ≥ 99% |
| **資料延遲** | 感測器發送至 DB 寫入的延遲 | P95 < 5 秒 |
| **設備感測器覆蓋率** | 已配置 MQTT 的設備比率 | ≥ 80% |

```sql
-- 每小時感測器資料完整率（以預期每分鐘 1 筆計算）
SELECT 
    equipment_id,
    sensor_type,
    strftime('%Y-%m-%d %H', recorded_at) as hour,
    COUNT(*) as received,
    60 as expected,
    ROUND(100.0 * COUNT(*) / 60, 1) as completeness_pct
FROM sensor_readings
WHERE recorded_at > datetime('now', '-24 hours')
GROUP BY equipment_id, sensor_type, hour
ORDER BY completeness_pct ASC;
```

### 5.2 告警品質指標

| 指標 | 定義 | 目標 |
|------|------|------|
| **MQTT 告警誤報率** | FP / 總告警數 | ≤ 20% |
| **告警去重率** | 相同設備相似告警合并率 | ≥ 90% |
| **閾值精準度** | 閾值設定合理性評分 | ≥ 3.5 / 5.0 |

## 6. KPI 量測方法

### 6.1 自動化量測腳本

```python
# scripts/kpi_report.py
# 每日 00:00 執行，輸出 KPI 報告至 syslog

async def generate_daily_kpi():
    report = {
        "date": datetime.utcnow().date().isoformat(),
        "system": {
            "api_p95_ms": await get_p95_response_time(),
            "error_rate_pct": await get_error_rate(),
            "mqtt_uptime_pct": await get_mqtt_uptime(),
        },
        "equipment": {
            "avg_vhs": await get_avg_vhs_score(),
            "low_vhs_count": await get_low_vhs_count(),
            "vlm_inspections": await get_vlm_inspection_count(),
        },
        "alerts": {
            "triggered": await get_alert_count(),
            "resolved": await get_resolved_alert_count(),
            "critical_unresolved": await get_critical_unresolved(),
        },
        "sensors": {
            "data_completeness_pct": await get_sensor_completeness(),
            "active_mqtt_configs": await get_active_mqtt_count(),
        }
    }
    
    await write_syslog(
        level="INFO",
        source="system",
        message="Daily KPI Report",
        extra=report
    )
    return report
```

### 6.2 KPI 資料來源對應

| KPI 類別 | 主要資料來源 | 備用資料來源 |
|----------|-------------|-------------|
| 系統效能 | `syslog.db` (duration_ms) | Docker health check |
| VHS 趨勢 | `vhs_readings` 表 | `equipment.vhs_score` |
| 告警效率 | `equipment_alerts` 表 | syslog (source=mqtt) |
| 感測器完整率 | `sensor_readings` 表 | MQTT message count |
| VLM 推論 | `inspection_reports` 表 | Pipeline status logs |
