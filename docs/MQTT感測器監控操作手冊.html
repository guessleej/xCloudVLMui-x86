模組概覽

02

頁面介面說明

UI Layout

頁面由上至下分為三個主要區塊：

① Broker 狀態列（最上方）

顯示 MQTT Broker 連線狀態、統計數字，以及操作按鈕列（自動刷新、刷新、測試發佈、新增設備）。

② 測試發佈面板（可展開）

點選「測試發佈」按鈕後展開，可手動輸入 Topic 與 Payload 向 Broker 發送測試訊息。預設隱藏。

③ 設備即時讀值（主要內容區）

以卡片排列顯示每台登錄設備的目前感測數值，每張卡片包含設備名稱、在線狀態與各類感測器最新讀值。

05

設備即時讀值

Device Cards

主要內容區，以網格形式展示每台已登錄設備的感測卡片。桌面版為 2～3 欄排列，每 5 秒自動刷新一次。

設備卡片說明

每張設備卡片包含以下資訊：

區域 內容
卡片頂列 設備名稱、設備 ID、安裝位置、在線狀態標籤（Online / Offline）、展開/收合鈕、刪除鈕
感測器讀值格 依感測器類型顯示圖示、中文標籤、最新數值（含單位）、資料品質標籤（good / warn / error）、更新時間（x 秒前）
最後回傳時間 卡片底部顯示最後一次收到資料的相對時間

在線狀態說明

Online（在線）
綠色邊框
設備最近有回傳資料，MQTT 訊息已成功接收
Offline（離線）
灰色邊框
設備尚未回傳資料，或已超過一段時間未收到訊息

資料品質標籤

標籤顏色說明
good綠色資料正常，感測器工作良好
warn黃色資料可能有異常或感測器有警告
error紅色資料異常，感測器回報錯誤狀態

展開 / 收合卡片

點選卡片右上角的 ∧ / ∨ 按鈕，可展開或收合感測器讀值區域。預設為展開狀態。收合後僅顯示設備標題列，方便同時監看大量設備。

提示 若設備已登錄但感測卡片顯示「尚未收到任何感測資料」，代表 MQTT 訊息尚未到達。請確認感測器 Topic 格式正確，並參考第 11 節 Topic 規範。
06

新增設備

Add Device

在監控頁面快速登錄新設備，登錄後設備卡片立即出現在頁面上，等待感測器開始回傳資料。

操作步驟

  1. 1
    點選「＋ 新增設備」按鈕

    位於頁面右上角，點選後彈出新增設備對話框。

  2. 2
    填寫設備 ID（必填）

    MQTT Topic 中的唯一識別碼。建議使用英文小寫加底線，例如：compressor_01。登錄後不可修改。

  3. 3
    填寫顯示名稱（必填）

    頁面上顯示的設備中文名稱，例如：壓縮機 #1。

  4. 4
    選擇設備類型與安裝位置（選填）

    設備類型：sensor / compressor / motor / pump / conveyor / cooler / panel / plc。安裝位置例如：廠房 A / 區域 1。

  5. 5
    填寫 Topic 前綴（必填）

    MQTT 訂閱主題的前綴，格式為 xcloud/{device_id}，例如：xcloud/compressor_01。

  6. 6
    點選「確認新增」

    儲存成功後對話框自動關閉，設備卡片出現在頁面上。

欄位必填說明範例
設備 ID必填MQTT Topic 識別碼，全系統唯一compressor_01
顯示名稱必填頁面顯示用名稱壓縮機 #1
設備類型選填設備分類，預設 sensorcompressor
安裝位置選填設備實體安裝位置廠房 A 區域 1
Topic 前綴必填MQTT 主題前綴xcloud/compressor_01
備註選填其他說明文字主線壓縮機
注意 設備 ID 一旦建立後無法直接修改。若需變更，請先刪除該設備再重新新增。如要修改顯示名稱、位置等其他資訊，請至「感測器管理」頁面操作。
07

刪除設備

Delete

在設備卡片上可直接刪除設備。

  1. 1
    點選卡片右上角的 🗑 刪除圖示

    將滑鼠移至設備卡片標題列右側，找到垃圾桶圖示。

  2. 2
    確認刪除對話框

    瀏覽器會彈出確認對話框，顯示「確定要刪除設備『XXX』？相關所有讀值也將一併刪除。」

  3. 3
    點選「確定」

    設備及其所有歷史感測資料將從資料庫中永久移除。

警告：此操作無法復原 刪除設備時，該設備所有歷史感測讀值、警報閾值設定將一併從資料庫中移除，且無法恢復。請謹慎操作。
08

測試發佈訊息

Test Publish

工程師可使用測試發佈功能，手動向 MQTT Broker 發送一則訊息，驗證整條資料管道是否正常運作（感測器 → Broker → Backend → 資料庫 → 頁面）。

操作步驟

  1. 1
    點選「測試發佈」按鈕

    按鈕位於頁面右上角操作列，點選後面板展開（橘色高亮表示已展開）。

  2. 2
    填寫 Topic

    格式：xcloud/{device_id}/{sensor_type}，例如：xcloud/compressor_01/temperature

  3. 3
    填寫 Payload

    支援兩種格式：
    JSON 格式：{"value": 72.5, "unit": "°C", "quality": "good"}
    純數字：72.5

  4. 4
    點選「發佈」按鈕

    訊息發送後約 1.5 秒，頁面自動刷新，若設備已登錄則可在卡片上看到剛發佈的數值。

Payload 格式說明

// 完整 JSON 格式（推薦） { "value": 72.5, // 感測數值（必填） "unit": "°C", // 單位（選填） "quality": "good" // 品質：good / warn / error（選填，預設 good） } // 簡易純數字格式 72.5
09

自動刷新控制

Auto Refresh
模式外觀行為
自動刷新中 按鈕橘色高亮，圖示旋轉 每 5 秒自動向後端 API 取得最新資料，頁面持續更新
手動模式 按鈕一般灰色 停止自動刷新，需手動點選「刷新」按鈕才會更新

點選「自動刷新中 / 手動模式」按鈕可切換兩種模式。頁面預設為自動刷新模式。

建議 監控室長期顯示時建議保持自動刷新模式。若需截圖存檔或詳細檢視某一筆資料時，可切換至手動模式暫停更新。

支援感測器類型

Sensor Types

系統支援以下感測器類型，不同類型在卡片上有對應的圖示與顏色區別：

類型（英文）中文名稱圖示顏色常用單位
temperature溫度🟠 橘色°C、°F
vibration振動🟣 紫色mm/s、g
pressure壓力🔵 藍色bar、kPa、psi
rpm轉速🩵 青色rpm
humidity濕度🩵 綠青色%RH
voltage電壓🟡 黃色V、kV
current電流🟡 琥珀色A、mA
power功率🔴 紅色W、kW
其他感測器類型 若感測器類型不在上表中（例如 flow、level 等），系統仍可接收並儲存資料，卡片上會以預設灰色圖示顯示，類型名稱直接作為標籤使用。
11

MQTT Topic 規範

Topic Spec

Topic 格式

xcloud/{device_id}/{sensor_type} // 範例 xcloud/compressor_01/temperature xcloud/motor_a/vibration xcloud/pump_02/pressure xcloud/cooling_fan/rpm
段落說明規則
xcloud固定前綴，所有設備皆使用固定，不可變更
{device_id}設備識別碼，對應登錄時填寫的「設備 ID」英文小寫、數字、底線，例：compressor_01
{sensor_type}感測器類型，決定圖示與顯示名稱英文小寫，例：temperature、vibration
注意 {device_id} 必須完全符合系統中已登錄的設備 ID，否則 Backend 收到訊息後無法對應到設備，資料將不會被儲存。

常見問題排查

Troubleshoot
問題現象可能原因解決方式
Broker 顯示「離線」 mqtt-broker 容器未啟動，或後端服務無法連線 確認 docker ps 中 mqtt-broker 容器狀態為 Up；重啟 xcloud-qa-backend 容器
設備卡片出現但無讀值 感測器 Topic 格式不正確，或設備 ID 對應錯誤 確認 Topic 格式為 xcloud/{device_id}/{sensor_type}，且 device_id 與登錄的完全一致
設備顯示 Offline 感測器未回傳資料，或網路中斷 確認感測器設備電源與網路正常；使用「測試發佈」功能手動發送測試訊息確認管道
累計訊息數不增加 感測器訂閱主題不在 xcloud/# 範圍內 確認感測器發布的 Topic 以 xcloud/ 開頭
頁面資料停止更新 自動刷新已被關閉，或網路連線中斷 確認「自動刷新中」按鈕為橘色高亮；手動點選「刷新」確認 API 是否正常回應
新增設備後找不到卡片 頁面尚未刷新 新增成功後頁面會自動刷新，若未出現請手動點選「刷新」按鈕
