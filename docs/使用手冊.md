## 1. 系統概述

### 1.1 產品簡介

**xCloud VLM Platform** 是由云碩科技（xCloud）開發，專為 **Advantech AIR-030 邊緣運算主機**設計的設備視覺監控與維修管理平台。系統結合邊緣推論（Edge Inference）、視覺語言模型（Vision Language Model）、本地知識庫檢索生成（RAG）與 MQTT 感應器整合，在不依賴雲端服務的前提下，實現工廠設備的即時異常偵測、智慧診斷、感應器監控與維護紀錄管理。

### 1.2 核心功能模組

| 功能模組 | 頁面路由 | 說明 |
|---------|---------|------|
| **設備健康總覽** | `/main/dashboard` | VHS 健康分數趨勢、警報事件管理、四段式巡檢管線狀態、設備詳情面板 |
| **視覺巡檢** | `/main/vlm` | WebRTC 即時影像串流，透過 Gemma 4 E4B 進行視覺分析診斷 |
| **MQTT 感應器監控** | `/main/mqtt` | 感應器即時數據（溫度、震動、壓力）、閾值警報、歷史圖表 |
| **MQTT 設備管理** | `/main/mqtt-devices` | 設備新增、編輯、刪除、Topic 管理 |
| **維修知識庫** | `/main/rag` | RAG 語意問答，從上傳文件中搜尋故障解決方案 |
| **知識文件管理** | `/main/knowledge` | 上傳 PDF/TXT/MD/CSV 文件並自動嵌入向量索引 |
| **維護報告** | `/main/reports` | 自動將 VLM 診斷結果轉換為結構化 Markdown 報告 |
| **事件中心** | `/main/events` | 系統操作日誌查詢、等級篩選、統計摘要 |
| **系統設定** | `/main/settings` | OCR 引擎、Embedding 模型、LLM 模型切換 |
| **多方式登入** | `/auth/login` | Microsoft Entra ID、Google、GitHub OAuth 及本地管理員帳號 |

### 1.3 技術亮點

- **完全離線運作**：模型推論、資料庫、向量索引均在本地執行，無需網際網路連線
- **128K Token 長上下文**：Gemma 4 E4B 支援 131,072 tokens，可處理完整維修手冊
- **OpenAI 相容 API**：透過 llama.cpp 提供 `/v1/chat/completions` 和 `/v1/embeddings` 標準介面
- **MQTT 感應器整合**：Eclipse Mosquitto Broker，支援溫度、震動、壓力等多種感測器類型
- **四段式巡檢管線**：視覺取像 → 邊緣推論 → 知識整合 → 維護輸出，全程狀態可觀測
- **即時警報系統**：VHS 分數 DB 持久化，支援 VLM 推論 / 人工輸入 / 歷史種子三種來源
- **事件中心**：全系統 HTTP 請求自動記錄至獨立 syslog.db，支援等級、模組、關鍵字篩選
- **Docker 容器化部署**：七個服務容器統一管理，一鍵啟動
- **生產打包工具**：`build-release.sh` 一鍵打包 Docker 映像為離線安裝包

### 2.2 服務清單

| 服務名稱 | Container | Port | 說明 |
|---------|-----------|------|------|
| `model-init` | `model-init` | — | 首次啟動自動從 HuggingFace 下載 GGUF 模型（完成後退出）|
| `llama-cpp` | `llama-cpp` | 8080 | Gemma 4 E4B 推論引擎，提供 OpenAI 相容 API |
| `vlm-webui` | `vlm-webui` | 8090 | WebRTC 即時視覺分析介面（ARM64 / JetPack 6.0）|
| `backend` | `xcloud-backend` | 8000 | FastAPI 中介層，負責 RAG、報告、儀表板、MQTT、設定 |
| `frontend` | `xcloud-frontend` | 3000 | Next.js 網頁儀表板 |
| `nginx` | `xcloud-nginx` | 80 | 反向代理，統一入口 |
| `mosquitto` | `mosquitto` | 1883 / 9001 | Eclipse Mosquitto MQTT Broker |

### 2.3 服務啟動順序

```
model-init   →（下載/確認模型）→ 退出 (exit 0)
llama-cpp    →（等待模型就緒）→（載入 128K context）→ healthy
backend      →（等待 llama-cpp healthy）→（初始化 DB / ChromaDB / MQTT）→ healthy
vlm-webui    →（等待 llama-cpp healthy）→（啟動 WebRTC Server）
frontend     →（等待 backend healthy）→（Next.js 啟動）
nginx        →（所有服務就緒）→（開始接受請求）
mosquitto    →（獨立啟動，不影響其他服務）
```

### 2.4 資料流說明

#### 視覺巡檢流程
```
USB/IP 相機 → VLM WebUI (WebRTC)
           → llama-cpp (Gemma 4 E4B 圖文推論)
           → 分析 JSON → Backend (FastAPI)
           → VhsReading 寫入 SQLite
           → Report 儲存為 Markdown
           → Frontend 儀表板顯示
```

#### MQTT 感應器流程
```
設備感應器 → Mosquitto Broker (Port 1883)
          → Backend MQTT Listener 訂閱 xcloud/#
          → MqttSensorReading 寫入 SQLite
          → 閾值判定 → EquipmentAlert 寫入
          → Frontend /mqtt 頁面輪詢展示
```

#### RAG 知識庫查詢流程
```
使用者提問 → Frontend → POST /api/rag/query
          → Backend → llama-cpp /v1/embeddings（問題向量化）
          → ChromaDB 語意搜尋（top_k 相關段落）
          → llama-cpp /v1/chat/completions（生成回答）
          → 回傳回答 + 來源引用 → Frontend 顯示
```

#### 四段式巡檢管線狀態
```
GET /api/dashboard/pipeline-status
→ asyncio.gather 平行檢查：
  ① 視覺取像：VLM WebUI HTTP 可達性
  ② 邊緣推論：llama-cpp /health + 模型名稱
  ③ 知識整合：ChromaDB 文件數 + RAG 文件數
  ④ 維護輸出：SQLite 警報數 + 報告數
→ 每 30 秒自動重新整理
```

```bash

# Docker version 24.0.x

docker compose version
# Docker Compose version v2.x.x

# 確認 NVIDIA Container Toolkit（AIR-030 必須）
nvidia-ctk --version
docker run --rm --runtime=nvidia \
  nvcr.io/nvidia/l4t-base:r36.3.0 nvidia-smi
```

### 5.3 OAuth 應用程式設定（可選）

若需使用 Microsoft / Google / GitHub OAuth 登入，請提前建立應用程式：

#### Microsoft Entra ID（Azure AD）
1. 登入 Azure Portal → **App registrations** → **New registration**
2. 名稱：`xCloud VLM Platform`
3. 支援帳戶類型：**Accounts in this organizational directory only**
4. 重新導向 URI：`http://<主機IP>/api/auth/callback/microsoft-entra-id`
5. 記錄：**Client ID**、**Client Secret**、**Tenant ID**

#### Google OAuth
1. 登入 Google Cloud Console → **APIs & Services** → **Credentials**
2. 建立 **OAuth 2.0 Client ID**，類型選 **Web application**
3. 授權重新導向 URI：`http://<主機IP>/api/auth/callback/google`
4. 記錄：**Client ID**、**Client Secret**

#### GitHub OAuth
1. 登入 GitHub → **Settings** → **Developer settings** → **OAuth Apps** → **New OAuth App**
2. Homepage URL：`http://<主機IP>`
3. Authorization callback URL：`http://<主機IP>/api/auth/callback/github`
4. 記錄：**Client ID**、**Client Secret**

## 6. 地端部署步驟

### 6.1 快速部署（建議方式）

```bash

cd xcloudvlmui

# 步驟一：完整初始化
# 依序執行：產生 SSL 憑證 → 複製 .env 範本 → 下載 GGUF 模型

# 步驟二：編輯環境變數（重要！）
# 請參考第 7 節填入正確設定
nano backend/.env
nano frontend/.env.local

# 步驟三：啟動所有服務
make up

# 步驟四：驗證服務健康狀態
make test
```

### 6.2 手動部署（逐步說明）

#### 步驟 1：產生 SSL 憑證

WebRTC 視覺巡檢需要 HTTPS 環境，請產生自簽憑證：

```bash
# 自動產生（透過腳本）
bash scripts/gen-ssl.sh

# 或手動產生
mkdir -p ssl
openssl req -x509 -newkey rsa:4096 \
  -keyout ssl/key.pem \
  -out ssl/cert.pem \
  -days 365 \
  -nodes \
  -subj "/C=TW/ST=Taiwan/L=Taipei/O=xCloud/CN=air-030.local"
```

#### 步驟 2：設定 Backend 環境變數

```bash
cp backend/.env.example backend/.env
nano backend/.env
```

**最小必要設定**：

```env
# ── 基礎安全 ──────────────────────────────────
SECRET_KEY=請換成32字元以上的隨機字串

# ── llama.cpp 推論引擎 ────────────────────────
LLM_BASE_URL=http://llama-cpp:8080
LLM_MODEL=gemma-4-e4b-it

# ── 資料存放路徑 ───────────────────────────────
DATABASE_URL=sqlite+aiosqlite:////data/xcloudvlm.db
CHROMA_PERSIST_DIR=/data/chroma
```

### 9.3 視覺巡檢

3. 巡檢完畢後，點擊右上角「**儲存為報告**」

#### 9.3.3 儲存為維護報告

點擊「**儲存為報告**」後：
1. 系統呼叫 `POST /api/reports/capture-vlm-session`

3. 報告儲存至 SQLite，並自動寫入 VHS 分數至 `vhs_readings` 表格

### 9.5 MQTT 設備管理

**訪問路徑**：側欄「**MQTT 設備**」（`/main/mqtt-devices`）

#### 9.5.1 新增設備

1. 點擊右上角「**+ 新增設備**」
2. 填入設備資訊：
   - **設備 ID**（唯一識別碼，如 `compressor_01`）
   - **設備名稱**（如「壓縮機 #1」）
   - **設備類型**（sensor / gateway / controller）
   - **位置**（如「A 廠房 / 產線 1」）
   - **MQTT Topic 前綴**（如 `xcloud/compressor_01`）
   - **描述**（選填）
3. 點擊「**儲存**」

#### 9.5.2 設定警報閾值

1. 點擊設備列表中的設備
2. 切換至「**閾值設定**」分頁
3. 新增感測器類型閾值：
   - 感測器類型（temperature / vibration / pressure / rpm）
   - 安全範圍：最小值 / 最大值
   - 警告範圍：warn_min / warn_max（介於安全範圍之間）
   - 單位（°C / m/s² / bar / RPM）
4. 超出閾值時，後端自動於 `equipment_alerts` 建立警報

#### 9.5.3 MQTT Topic 規範

| Topic 格式 | 說明 |
|-----------|------|
| `xcloud/{device_id}/temperature` | 溫度讀值 |
| `xcloud/{device_id}/vibration` | 震動讀值 |
| `xcloud/{device_id}/pressure` | 壓力讀值 |
| `xcloud/{device_id}/rpm` | 轉速讀值 |
| `xcloud/{device_id}/status` | 設備狀態 JSON |

**Payload 格式**（JSON）：
```json
{
  "value": 75.3,
  "unit": "°C",
  "quality": "good",
  "timestamp": "2026-04-10T08:30:00Z"
}
```

- 清除對話按鈕（垃圾桶圖示）可重設對話記錄

### 9.7 知識文件管理

## 12. 生產打包與部署

### 12.1 概述

`scripts/build-release.sh` 是生產級 Docker 映像打包工具，可將所有服務映像打包為離線安裝包，便於部署至沒有網際網路的工廠 AIR-030 主機。

**輸出結構**：
```
dist/xcloudvlmui-v{VERSION}-{DATE}/

│   ├── xcloudvlmui-backend-{VERSION}.tar.gz
│   ├── xcloudvlmui-frontend-{VERSION}.tar.gz
│   ├── xcloudvlmui-vlm-webui-{VERSION}.tar.gz   # --skip-vlm 時省略
│   ├── nginx-1.25-alpine.tar.gz
│   └── eclipse-mosquitto-2.tar.gz                # --skip-mqtt 時省略
├── config/
│   ├── docker-compose.yml
│   ├── nginx/nginx.conf
│   ├── backend/.env.example
│   └── frontend/.env.local.example
├── scripts/
│   ├── install.sh                                 # 八步驟一鍵安裝腳本
│   ├── download-model.sh
│   ├── gen-ssl.sh
│   └── test-services.sh
├── Makefile
├── README.md
└── VERSION
```

### 12.2 打包指令

```bash
# 進入專案目錄
cd xcloudvlmui

# 方式一：使用 Makefile（建議）
make release          # ARM64 快速打包（跳過 vlm-webui）
make release-full     # ARM64 完整打包（含 vlm-webui，約需 30–60 分鐘）
make release-amd64    # x86_64 打包（本機 x86 測試用）

# 方式二：直接執行腳本
bash scripts/build-release.sh [OPTIONS]
```

**腳本選項**：

| 選項 | 說明 | 預設值 |
|-----|------|-------|
| `-v, --version VERSION` | 指定版本號 | 從 git tag 或 `1.0.0` |
| `-p, --platform PLATFORM` | 目標平台 | `linux/arm64` |
| `--skip-vlm` | 跳過 vlm-webui 映像建置 | 不跳過 |
| `--skip-mqtt` | 跳過 mosquitto 映像拉取 | 不跳過 |
| `--no-cache` | 強制重新建置（不使用快取）| 使用快取 |
| `--push REGISTRY` | 打包後推送至 Registry | 不推送 |

**範例**：
```bash
# 指定版本號，跳過 vlm-webui
bash scripts/build-release.sh --version 1.1.0 --skip-vlm

# 完整打包含推送
bash scripts/build-release.sh --version 1.1.0 --push registry.example.com/xcloud

# x86 測試打包
bash scripts/build-release.sh --platform linux/amd64 --skip-vlm
```

### 13.6 常見錯誤代碼

| HTTP 狀態碼 | 常見情境 | 解決方式 |
|-----------|---------|---------|
| 422 | 請求 Body 格式錯誤（Pydantic 驗證失敗）| 確認欄位名稱與型別（參考 `/docs`）|
| 503 | 後端服務未就緒 | 等待服務啟動，執行 `make test` 確認 |
| 502 | Nginx 代理失敗 | 確認目標服務容器正在運行 |
| 500 | 後端內部錯誤 | 查看 `make logs-backend` |
| 404 | 路由不存在 | 確認 API 路徑（參考 `/docs`）|
| 401 | 未授權（token 無效或過期）| 重新登入，確認 AUTH_SECRET 設定 |

## 14. 安全性建議

### 14.1 部署前必要設定清單

- [ ] **修改管理員密碼**：將 `ADMIN_PASSWORD=admin123` 更換為強密碼（12+ 字元）
- [ ] **更換 AUTH_SECRET**：`openssl rand -base64 32`（前後端需一致）
- [ ] **更換 SECRET_KEY**：Backend 安全金鑰同上
- [ ] **設定 HTTPS**：正式環境使用有效 TLS 憑證（Let's Encrypt 或企業 CA）
- [ ] **限制 ALLOWED_ORIGINS**：僅允許實際前端來源
- [ ] **限制 API 存取**：不建議將 Port 8080（llama.cpp）直接對外開放

### 14.2 建議設定

- [ ] **啟用 OAuth 登入**：停用或限制本地帳號，改用 Microsoft/Google OAuth
- [ ] **MQTT 認證**：在 `mosquitto.conf` 啟用使用者名稱 / 密碼認證
- [ ] **定期備份**：每日備份 SQLite DB 與 ChromaDB（建議排程 cron）
- [ ] **日誌保留**：定期清除過期 syslog（`DELETE /api/syslog/?before_days=30`）
- [ ] **系統更新**：定期更新 Docker Base Image 與相依套件

### 14.3 網路隔離建議

```
[使用者網段] ──→ Port 80/443 ──→ [Nginx] ──→ [Docker 內網]
[MQTT 設備] ──→ Port 1883  ──→ [Mosquitto]
                                             ↑
                      僅開放 80/443 與 1883 給外部
                      8080/8000/8090 僅限內網存取
```

### 附錄 D：MQTT Topic 規範

| Topic 格式 | 感測器類型 | Payload 範例 |
|-----------|----------|-------------|
| `xcloud/{id}/temperature` | 溫度 | `{"value": 75.3, "unit": "°C", "quality": "good"}` |
| `xcloud/{id}/vibration` | 震動 | `{"value": 2.1, "unit": "m/s²", "quality": "good"}` |
| `xcloud/{id}/pressure` | 壓力 | `{"value": 8.5, "unit": "bar", "quality": "good"}` |
| `xcloud/{id}/rpm` | 轉速 | `{"value": 1450, "unit": "RPM", "quality": "good"}` |
| `xcloud/{id}/status` | 設備狀態 | `{"online": true, "uptime_s": 3600}` |

`quality` 可選值：`good`（正常）/ `stale`（資料過舊）/ `error`（感測器故障）

### 附錄 E：VHS 健康分數定義

| 分數範圍 | 健康區間 | 說明 | 建議動作 |
|---------|---------|------|---------|
| 70 – 100 | 🟢 穩定區 | 設備運作正常 | 按計畫維保即可 |
| 40 – 69 | 🟡 警戒區 | 有輕微退化跡象 | 安排近期檢修，加強監控頻率 |
| 0 – 39 | 🔴 危急區 | 異常明顯，風險高 | 立即停機檢修，避免重大損失 |

**VHS 分數來源優先順序**（同一天多筆時，以最高優先度為準）：
1. `vlm`（VLM 視覺推論）
2. `manual`（人工輸入）
3. `seed`（歷史種子資料）
4. `estimated`（系統估算，不顯示圓點）
