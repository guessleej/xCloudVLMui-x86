## 2. Sprint 回顧流程（Retrospective）

### 執行頻率

**每 2 週一次**（Sprint 結束後）

### 回顧會議結構（60 分鐘）

```
[00–10] 開場：回顧上次改進行動落實情況
[10–30] 三個象限回顧：
         ✅ 做得好的（Keep）
         🔧 需要改進的（Improve）
         🆕 下次嘗試的（Try）
[30–50] 根因分析（問題項目）
[50–60] 確定改進行動項目（≤ 3 項）
```

## 4. 技術債務登錄表

| 債務 ID | 描述 | 引入版本 | 影響 | 優先級 | 計畫償還版本 | 狀態 |
|--------|------|---------|------|--------|------------|------|
| **TD-001** | `/api/dashboard/*` 廢棄 shim（`routers/dashboard.py`）| v1.1.0 | 維護複雜度增加 | 低 | v1.3.0 | 🔄 已記錄 |
| **TD-002** | `/api/rag/*` 廢棄 shim（`routers/rag.py`）| v1.1.0 | 維護複雜度增加 | 低 | v1.3.0 | 🔄 已記錄 |
| **TD-003** | 測試覆蓋率目前 < 70%（僅 4 個測試檔案）| v1.1.0 | CI 品質門檻未滿足 | 🔴 高 | v1.2.0 | 🔄 進行中 |

| **TD-005** | Equipment 資料為靜態 Mock，無正式 DB 資料表 | v1.0.0 | 資料持久化缺失 | 🔴 高 | v1.2.0 | 🔄 規劃中 |
| **TD-006** | MQTT 未加密傳輸（R011）| v1.0.0 | 安全風險 | 中 | v1.2.0 | 🔄 規劃中 |
| **TD-007** | ChromaDB 無定期備份自動化 | v1.0.0 | R007 (索引損毀) | 中 | v1.2.0 | 🔄 規劃中 |
| **TD-008** | 前端 API 呼叫部分仍指向舊 `/api/dashboard/*` 端點 | v1.1.0 | 廢棄 API 依賴 | 中 | v1.2.0 | 🔄 待遷移 |

### 技術債務管理原則

1. **記錄優先**：每引入一項技術債務，立刻加入此表 + GitHub Issue（label: `tech-debt`）
2. **量化影響**：每項債務標明影響等級（高/中/低）
3. **計畫償還**：高優先級債務必須在下一版本償還
4. **每版本審查**：發布前確認哪些債務已償還，更新此表

## 5. 改進 Backlog 追蹤機制

### GitHub Issues 標籤體系

| Label | 顏色 | 說明 | 觸發來源 |
|-------|------|------|---------|
| `improvement` | 🟢 綠 | 一般改進項目 | Retrospective |
| `kpi-alert` | 🔴 紅 | KPI 未達標觸發 | KPI 報告 |
| `tech-debt` | 🟡 黃 | 技術債務 | 開發過程中記錄 |
| `post-release` | 🟠 橙 | Post-Release 發現 | Post-Release Review |
| `security` | 🔴 紅 | 安全性改進 | 安全審查 |
| `performance` | 🔵 藍 | 效能改進 | KPI / 監控 |

### 改進優先級矩陣（ICE 評分）

```
優先級分數 = Impact × Confidence × Ease
  Impact    (1–10): 對業務/技術目標的影響程度
  Confidence(1–10): 我們對改進效果的確定程度
  Ease      (1–10): 實作難度（10 = 很容易）

分數 ≥ 500 → 下一個 Sprint 必做
分數 200-499 → 排入未來 Sprint Backlog
分數 < 200  → 記錄但暫不排程
```

### Backlog 範本

```markdown
## 改進項目：[標題]

**Labels:** improvement / kpi-alert / tech-debt
**觸發來源：** Retrospective Sprint #XX / KPI 報告 YYYY-MM-DD

<!-- 什麼問題？影響是什麼？ -->

### 期望改進後的狀態
<!-- 改進後 KPI 預期變化？ -->

### 建議方案
<!-- 技術實作建議 -->

### ICE 評分
- Impact: X/10 (說明)
- Confidence: X/10 (說明)
- Ease: X/10 (說明)
- **總分: XXX**

### 驗收標準
- [ ] 
- [ ]

- RISK_MATRIX.md: R-XXX
- KPI: M-XXX
```

## 7. 知識沉澱機制

### 7.1 決策記錄（ADR）

每當做出重要技術決策，**必須**在 `docs/adr/` 建立 ADR 文件。

**ADR 觸發場景：**
- 選擇或替換核心技術（資料庫、框架、AI 模型）
- 重大 API 設計變更
- 有多個方案選擇時的決策理由
- 技術債務的「暫時接受」決策

**ADR 命名規則：**
```
docs/adr/ADR-00X-{short-description}.md
例：ADR-005-mqtt-tls-encryption.md
```

**現有 ADR：**
- [ADR-001](adr/ADR-001-sqlite-over-postgresql.md) — SQLite 優於 PostgreSQL（v1.x）
- [ADR-002](adr/ADR-002-llama-cpp-over-vllm.md) — llama.cpp 優於 vLLM（ARM64）
- [ADR-003](adr/ADR-003-chromadb-over-pgvector.md) — ChromaDB 優於 pgvector
- [ADR-004](adr/ADR-004-mqtt-sensor-integration.md) — MQTT 優於 HTTP Polling

### 7.2 Runbook 維護

重大故障後，更新 `docs/OBSERVABILITY.md` 的對應章節：
- 新增告警規則（§6）
- 更新診斷查詢（§7.2）
- 更新緊急處理步驟（§9）

### 7.3 CHANGELOG 維護

所有值得記錄的變更，即時更新 `CHANGELOG.md` 的 `[Unreleased]` 區段：
- 新功能 → `### Added`
- 修正 → `### Fixed`
- 效能改善 → `### Changed`
- 技術債務償還 → `### Changed`
- 安全修補 → `### Security`

### 8.2 改進投資報酬率（ROI）追蹤

```
改進 ROI 計算：
  ROI = (改進後 KPI 值 - 改進前 KPI 值) / 投入工程師日 × 100%

例：
  CI/CD 全自動化：
    改進前：手動驗證 PR 需 2 小時
    改進後：自動化 20 分鐘
    節省：1.67 小時/PR × 預估 20 PR/月 = 33.4 小時/月
    投入：2 工程師日 = 16 小時
    ROI = 33.4 / 16 × 100% = 208.75% / 月
```
