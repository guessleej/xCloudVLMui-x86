## 1. 開發環境設置

### 1.1 必要工具

| 工具 | 版本要求 | 用途 |
|------|----------|------|
| Docker Desktop | ≥ 4.30 | 本機容器環境 |
| Docker Buildx | ≥ 0.14 | 多平台映像建置 |
| Python | 3.11+ | Backend 開發 |
| Node.js | 20 LTS | Frontend 開發 |
| Make | 任意 | Makefile 自動化 |
| Git | ≥ 2.40 | 版本控制 |

### 1.2 首次設置步驟

```bash
# 1. Clone 儲存庫
git clone https://github.com/guessleej/xCloudVLMui.git
cd xCloudVLMui

# 2. 初始化環境（SSL + .env + 模型下載）
make setup

# 3. 啟動開發模式（熱重載，CPU 模式）
make dev

# 4. 確認服務健康狀態
make test
```

### 1.3 Backend 本機開發

```bash
cd backend

# 建立虛擬環境
python -m venv .venv
source .venv/bin/activate  # macOS/Linux

# 安裝依賴
pip install -r requirements.txt
pip install -r requirements-dev.txt

# 啟動開發伺服器（自動重載）
uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

### 1.4 Frontend 本機開發

```bash
cd frontend

# 安裝依賴
npm install

# 複製環境變數
cp .env.local.example .env.local
# 填入 NEXT_PUBLIC_API_URL 等設定

# 啟動開發伺服器
npm run dev
```

## 2. Git 工作流程

### 2.1 分支策略（Git Flow 簡化版）

```
main ──────────────────────────────── 穩定生產版本
  │
  ├── develop ────────────────────── 整合開發分支
  │     │
  │     ├── feature/feat-xxx ──────── 功能開發
  │     ├── fix/bug-xxx ──────────── 錯誤修復
  │     └── refactor/xxx ─────────── 重構
  │
  └── hotfix/xxx ──────────────────── 緊急修復（直接從 main 分支）
```

### 2.2 分支命名規則

| 類型 | 格式 | 範例 |
|------|------|------|
| 功能 | `feature/功能描述` | `feature/mqtt-threshold-alert` |
| 修復 | `fix/問題描述` | `fix/vhs-calculation-error` |
| 重構 | `refactor/描述` | `refactor/pipeline-service` |
| 文件 | `docs/描述` | `docs/update-api-reference` |
| 緊急 | `hotfix/描述` | `hotfix/mqtt-connection-leak` |

### 2.3 Commit 訊息規範（Conventional Commits）

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

**Type 類型：**

| Type | 說明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 錯誤修復 |
| `docs` | 文件更新 |
| `style` | 程式碼格式（不影響邏輯） |
| `refactor` | 重構（非新功能、非修復） |
| `test` | 測試相關 |
| `chore` | 建置/工具/CI 更新 |
| `perf` | 效能優化 |

**Scope 範例：** `backend`, `frontend`, `mqtt`, `pipeline`, `vhs`, `docker`

**範例：**

```bash
feat(mqtt): add threshold-based alert generation
fix(vhs): correct 14-day trend window calculation
docs(api): update pipeline-status endpoint spec
chore(docker): upgrade llama.cpp to r36.4-tegra
```

## 3. 程式碼規範

### 3.1 Python（Backend）

**工具鏈：**
- **Formatter：** Black（行長 100）
- **Linter：** Ruff
- **Type checker：** mypy（strict mode）
- **Import 排序：** isort（透過 Ruff）

**設定（`pyproject.toml`）：**

```toml
[tool.black]
line-length = 100
target-version = ["py311"]

[tool.ruff]
line-length = 100
select = ["E", "F", "I", "UP", "B", "C4"]
ignore = ["E501"]

[tool.mypy]
python_version = "3.11"
strict = true
ignore_missing_imports = true
```

**命名規則：**

| 元素 | 規則 | 範例 |
|------|------|------|
| 變數/函式 | snake_case | `get_equipment_by_id` |
| 類別 | PascalCase | `VHSService` |
| 常數 | UPPER_SNAKE | `SOURCE_PRIORITY` |
| 私有屬性 | `_` 前綴 | `_db_session` |

**API 端點設計原則：**

```python
# ✅ 正確：使用 async/await、依賴注入、Pydantic schema
@router.get("/{equipment_id}", response_model=EquipmentResponse)
async def get_equipment(
    equipment_id: int,
    db: AsyncSession = Depends(get_db),
    service: EquipmentService = Depends(get_equipment_service),
) -> EquipmentResponse:
    return await service.get_by_id(equipment_id)

# ❌ 避免：同步函式、原始 SQL、裸 dict 回傳
@router.get("/{id}")
def get_equipment(id: int):
    conn = sqlite3.connect("db.sqlite")
    return {"id": id, ...}
```

### 3.2 TypeScript（Frontend）

**工具鏈：**
- **Formatter：** Prettier（`printWidth: 100`）
- **Linter：** ESLint + `@typescript-eslint`
- **型別：** TypeScript strict mode

**設定（`.eslintrc.json`）：**

```json
{
  "extends": ["next/core-web-vitals", "plugin:@typescript-eslint/recommended"],
  "rules": {
    "@typescript-eslint/no-explicit-any": "error",
    "@typescript-eslint/explicit-function-return-type": "warn",
    "no-console": ["warn", { "allow": ["warn", "error"] }]
  }
}
```

**元件規則：**

```typescript
// ✅ 正確：具名函式元件、明確 Props 型別
interface EquipmentCardProps {
  equipment: Equipment
  onSelect: (id: number) => void
}

export function EquipmentCard({ equipment, onSelect }: EquipmentCardProps) {
  return <div onClick={() => onSelect(equipment.id)}>{equipment.name}</div>
}

// ❌ 避免：匿名元件、any 型別、useEffect 依賴不完整
export default ({ data }: any) => <div>{data}</div>
```

## 5. Pull Request 流程

### 5.1 PR 建立規則

1. **分支保護：** `main` 和 `develop` 為保護分支，不可直接 push
2. **最小評審：** 至少 1 位 Reviewer 核准
3. **CI 必須通過：** 所有 GitHub Actions 檢查必須 ✅
4. **描述完整：** 使用 PR 模板填寫說明

### 5.2 PR 模板

```markdown
## 變更摘要
<!-- 簡述此 PR 做了什麼 -->

## 變更類型
- [ ] feat：新功能
- [ ] fix：錯誤修復
- [ ] docs：文件更新
- [ ] refactor：重構

## 測試
- [ ] 已新增單元測試
- [ ] 已通過本機測試
- [ ] 已更新相關文件

## 截圖（UI 變更需提供）
<!-- 附上 Before/After 截圖 -->

## 注意事項
<!-- Reviewer 需特別注意的地方 -->
```

### 5.3 Code Review 清單

**Reviewer 需確認：**
- [ ] 邏輯正確性（邊界條件、錯誤處理）
- [ ] 安全性（SQL Injection、XSS、敏感資料外露）
- [ ] 效能（N+1 查詢、不必要的重渲染）
- [ ] 可讀性（命名清晰、適當註解）
- [ ] 測試覆蓋（新功能是否有對應測試）
- [ ] API 相容性（是否為破壞性變更）

## 6. 版本發布流程

### 6.1 語意化版本（SemVer）

```
MAJOR.MINOR.PATCH

MAJOR：不相容的 API 變更
MINOR：向下相容的新功能
PATCH：向下相容的錯誤修復
```

### 6.2 發布步驟

```bash
# 1. 確認 develop 分支測試全通過
git checkout develop
make test

# 2. 建立 release 分支
git checkout -b release/v1.2.0

# 3. 更新版本號（package.json、pyproject.toml、docs）
# 手動更新版本號

# 4. 建置並測試生產映像
make release-amd64    # 本機 x86 測試
make test

# 5. 合併至 main
git checkout main
git merge --no-ff release/v1.2.0

# 6. 打 Tag
git tag -a v1.2.0 -m "Release v1.2.0: Add MQTT alert thresholds"
git push origin main --tags

# 7. 建置生產 ARM64 映像（Jetson 部署用）
make release    # 跳過 vlm-webui（速度快）
# 或
make release-full  # 含 vlm-webui

# 8. 合併回 develop
git checkout develop
git merge --no-ff main
```

### 6.3 Changelog 格式

```markdown
# Changelog

## [v1.2.0] - 2026-04-10

### Added
- MQTT 感測器閾值超限自動告警
- Equipment Detail Drawer 中的感測器即時讀值面板
- syslog.db 事件中心分頁查詢

### Changed
- VHS 趨勢圖改為 14 天滾動窗口
- Pipeline 四段式狀態改為 asyncio.gather 並發執行

### Fixed
- 修復 VHS score 在無資料時回傳 null 的問題
- 修復 MQTT reconnect 時記憶體洩漏

### Security
- 更新 llama.cpp 至 r36.4-tegra，修復 CVE-2025-XXXX
```
