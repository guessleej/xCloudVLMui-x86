## 1. 風險評估方法

### 1.1 風險等級計算

```
風險評分 = 發生機率 (P) × 影響程度 (I)

發生機率 (P)        影響程度 (I)
5 - 幾乎確定        5 - 災難性（專案失敗）
4 - 很可能          4 - 重大（主要目標受損）
3 - 可能            3 - 中等（次要目標受損）
2 - 不太可能        2 - 輕微（可接受延誤）
1 - 極不可能        1 - 微不足道（可忽略）
```

### 1.2 風險矩陣圖

```
影響 │                  風險熱度圖
     │
  5  │  5   10  [15]  [20]  [25]
     │
  4  │  4    8   12   [16]  [20]
     │
  3  │  3    6    9    12   [15]
     │
  2  │  2    4    6     8    10
     │
  1  │  1    2    3     4     5
     └──────────────────────────── 機率
          1    2    3     4    5

[XX] = 高風險（≥15）  需立即處理
      = 中風險（8-12） 需監控
      = 低風險（≤6）   可接受
```

## 3. 高風險項目行動計畫

### 🔴 S-01：管理員預設密碼未更改（評分 16）

```
觸發條件：  ADMIN_PASSWORD=admin123 未在部署前更換
立即行動：  install.sh 部署流程強制詢問新密碼
監控方式：  首次登入後 UI Banner 提示更換密碼
驗收標準：  部署 Checklist 100% 完成，無 admin123 殘留
負責人：    DevOps 工程師
期限：      v1.2.0 納入部署腳本自動化
```

### 🔴 S-02：安全金鑰未更換（評分 15）

```
觸發條件：  AUTH_SECRET / SECRET_KEY 使用預設值
立即行動：  install.sh 自動執行 openssl rand -base64 32 生成金鑰
監控方式：  啟動時後端偵測 SECRET_KEY 是否為預設值並警告
驗收標準：  自動化測試確認預設值不通過啟動
負責人：    DevOps 工程師
期限：      v1.2.0
```

### 🟡 T-01：llama.cpp API 不相容（評分 12）

```
觸發條件：  上游 llama.cpp 版本升級破壞 /v1/chat/completions 介面
立即行動：  docker-compose.yml 鎖定映像 tag（不使用 latest）
監控方式：  每季一次 /v1/models 相容性測試
驗收標準：  升級前完整 regression test 通過
負責人：    AI 工程師
```

### 🟡 D-02：HuggingFace 無法連線（評分 12）

```
觸發條件：  工廠離線環境無法下載 Gemma 4 E4B GGUF 模型
立即行動：  文件化離線複製 SOP（USB / NFS 方式）
監控方式：  model-init 容器啟動日誌檢查
驗收標準：  離線部署測試通過
負責人：    DevOps 工程師
```

## 4. 風險監控機制

### 4.1 定期回顧

| 頻率 | 活動 | 參與者 |
|------|------|-------|
| 每日 | Daily Standup 阻礙回報 | 開發團隊 |
| 雙週 | Sprint 回顧，風險狀態更新 | 全團隊 |

### 4.2 風險狀態追蹤

| 狀態 | 說明 |
|------|------|
| 🔍 **識別** | 已記錄，尚未評估 |
| 📋 **評估** | 已評分，應對策略制定中 |
| 🛡️ **監控** | 應對策略已實施，持續觀察 |
| ✅ **已關閉** | 風險已消除或已接受 |
| 🔥 **已發生** | 風險已轉變為問題，啟動應急計畫 |
